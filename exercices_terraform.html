Contexte & Architecture cible

Le projet ShopMicro est une application e-commerce décomposée en quatre services indépendants. Votre objectif est de déployer et gérer cette architecture localement en utilisant Terraform avec le provider Docker, sans toucher à Docker Compose une fois la Partie 1 terminée.

Partie 1 — Docker : maîtrise de l'existant

01Lecture du code source

Avant toute chose, lisez les trois fichiers src/index.js.

  1. Dressez le tableau exhaustif de toutes les routes HTTP exposées par chaque service (méthode, chemin, comportement attendu).
  2. Le order-service appelle les deux autres services. Identifiez exactement quels appels sont faits, dans quel ordre, et ce qui se passe si l'un d'eux échoue avec un statut 404.
  3. Chaque service a deux modes de fonctionnement selon la présence ou non de DATABASE_URL. Expliquez la différence de comportement et pourquoi ce pattern est utile pendant le développement.

1. Routes HTTP :

  • user-service : GET /health, GET /users, GET /users/:id, POST /users, DELETE /users/:id
  • product-service : GET /health, GET /products, GET /products/:id, POST /products, PATCH /products/:id/stock, DELETE /products/:id
  • order-service : GET /health, GET /orders, GET /orders/:id, POST /orders

2. Appels inter-services dans POST /orders :

  1. GET user-service/users/:user_id → si 404 : renvoie immédiatement 404 « Utilisateur ou produit introuvable »
  2. Pour chaque item : GET product-service/products/:product_id → si 404 : même réponse
  3. Si tout OK : INSERT commande en BDD et retourne 201

3. Modes de fonctionnement : Avec DATABASE_URL → Pool PostgreSQL, données persistantes. Sans → tableaux JS en mémoire (memUsers, etc.), données perdues au redémarrage. Utile en dev : permet de tester les routes sans avoir PostgreSQL de lancé.

  2. Sans utiliser le navigateur, créez un utilisateur, un produit, puis une commande via curl. Écrivez les commandes utilisées.
  3. Tentez de créer une commande avec un user_id inexistant. Quel code HTTP obtenez-vous et pourquoi ?
  4. Stoppez uniquement le conteneur user-service. Que se passe-t-il si vous tentez de créer une commande ? Relancez le service et expliquez pourquoi restart: unless-stopped ne suffit pas dans ce cas.

Commandes curl :

curl -X POST http://localhost:3001/users -H "Content-Type: application/json" \
  -d '{"name":"Alice","email":"alice@test.com"}'
curl -X POST http://localhost:3002/products -H "Content-Type: application/json" \
  -d '{"name":"Laptop Pro","price":999.99,"stock":10}'
curl -X POST http://localhost:3003/orders -H "Content-Type: application/json" \
  -d '{"user_id":1,"items":[{"product_id":1,"quantity":2}]}'

user_id inexistant : Code HTTP 404 — l'order-service appelle GET /users/:id sur le user-service qui répond 404, et propage cette erreur.

user-service stoppé : POST /orders retourne 500 (axios → ECONNREFUSED). restart: unless-stopped redémarre après un crash interne, pas après un docker stop manuel. De plus, la dépendance de démarrage n'est vérifiée qu'au lancement initial de Compose.

03Analyse des Dockerfiles et optimisation
  1. Examinez les trois Dockerfiles. Quelle est la différence entre npm install et npm ci ? Pourquoi l'un est-il préféré dans un contexte d'image de production ?
  2. Lancez docker images | grep shopmicrfo (après build via Docker Compose). Notez les tailles. Proposez et implémentez une modification des Dockerfiles qui réduit la taille des images. Mesurez l'impact.
  3. Ajoutez un fichier .dockerignore dans chaque dossier de service. Quels fichiers doit-il exclure et pourquoi ?
  4. La directive HEALTHCHECK est présente dans les Dockerfiles. Quelle est la différence entre l'état healthy, unhealthy et starting d'un conteneur ? Comment le vérifier ?

1. npm install vs npm ci : npm ci installe exactement ce qui est dans package-lock.json (reproductible, échoue si le lock est désynchronisé). npm install peut modifier le lock. En production, npm ci garantit des builds reproductibles.

2. Optimisation taille : Ajout de && npm cache clean --force après npm ci dans les Dockerfiles → supprime le cache npm (~50–80 MB) et réduit la taille de l'image. Mesuré avec docker images | grep shopmicrfo avant/après.

3. .dockerignore : Exclure node_modules/ (réinstallé dans l'image), .git/, .env (secrets), *.log, Dockerfile, .dockerignore, *.md. Réduit le contexte de build et évite de copier des secrets dans l'image.

4. États HEALTHCHECK : starting = pendant le start_period, les échecs ne font pas basculer ; healthy = tous les checks passent ; unhealthy = retries échecs consécutifs. Vérification : docker inspect --format='{{.State.Health.Status}}' <container>

04Réseaux Docker
  1. Identifiez le réseau Docker créé par Docker Compose pour ce projet. Quel driver est utilisé ? Quel sous-réseau est assigné ?
  2. Exécutez un shell dans le conteneur order-service. Pouvez-vous pinguer le conteneur postgres par son nom ? Par son IP ? Expliquez le mécanisme DNS interne de Docker.
  3. Dans docker-compose.yml, les services n'ont pas de configuration networks explicite. Comment Docker Compose détermine-t-il le réseau à créer ?
  4. Créez manuellement un réseau Docker monitoring_net. Ajoutez dans le docker-compose.yml un service Adminer (interface web PostgreSQL) sur le port 8081, connecté à la fois au réseau de l'application et à monitoring_net.

1. Réseau Docker Compose : Réseau terraform-esgi-3-iw_default créé automatiquement, driver bridge, sous-réseau assigné dynamiquement (ex : 172.18.0.0/16).

2. DNS interne Docker : Oui, ping par nom fonctionne (ping postgres depuis order-service). Docker Compose assigne le nom de service comme alias DNS. Un résolveur DNS embarqué (127.0.0.11) dans chaque conteneur traduit les noms en IPs réelles.

3. Réseau implicite : Sans configuration networks explicite, Docker Compose crée automatiquement <répertoire_projet>_default et connecte tous les services à ce réseau.

4. Adminer ajouté : docker network create monitoring_net puis ajout dans docker-compose.yml :

adminer:
  image: adminer:latest
  ports: ["8081:8080"]
  networks: [default, monitoring_net]
  restart: unless-stopped
networks:
  monitoring_net:
    external: true

  1. Dans le dossier terraform/, listez et décrivez brièvement le rôle de chaque fichier .tf.
  2. Lancez terraform init. Que télécharge cette commande ? Où les fichiers sont-ils placés ?

  4. Lancez terraform plan. Relevez : combien de ressources vont être créées ? Quels sont les types de ressources distincts ? Y a-t-il des ressources qui dépendent d'autres ? Comment le distinguer dans la sortie du plan ?

1. Fichiers .tf : main.tf (provider), variables.tf (inputs), locals.tf (valeurs calculées), images.tf (images Docker), network.tf (réseau), postgres.tf (BDD), services.tf (conteneurs), volumes.tf (volumes), outputs.tf (exports), terraform.tfvars (valeurs des variables).

2. terraform init : Télécharge le provider kreuzwerker/docker depuis registry.terraform.io dans .terraform/providers/. Génère .terraform.lock.hcl avec les hash du provider.

3. terraform validate : Valide la syntaxe HCL et les types sans contacter Docker. Résultat : « Success! The configuration is valid. »

4. terraform plan : ~15 ressources (images build/pull, conteneurs, réseau, volumes). Types distincts : docker_image, docker_container, docker_network, docker_volume. Dépendances visibles dans l'ordre de création affiché (ex : conteneurs après leur image et le réseau).

  4. Lancez terraform plan une seconde fois sans rien changer. Que constatez-vous ? Expliquez pourquoi.

1. Ordre d'apparition : Postgres apparaît en premier (réseau + volume créés avant), puis les microservices (user, product), ensuite order-service (dépend de user et product), enfin le frontend. Correspond aux dépendances déclarées.

2. terraform output : Affiche les URLs de chaque service. Application vérifiée via curl sur les ports exposés.

3. terraform.tfstate : Chaque ressource a instances[].attributes avec tous les attributs Docker réels (ID SHA256 du conteneur, IP assignée, network_settings, created_at). Info cruciale absente du code .tf : l'ID unique Docker permettant à Terraform de retrouver la ressource réelle.

4. Second plan : « No changes. Your infrastructure matches the configuration. » — Terraform compare le state et l'infra réelle, aucune différence. Principe d'idempotence.

07Variables, locals et expressions
  1. Dans variables.tf, la variable db_password est marquée sensitive = true. Modifiez sa valeur via la ligne de commande (-var) et observez la différence dans le plan. Que cache Terraform exactement ?
  2. Ajoutez une variable restart_policy de type string avec une contrainte de validation n'acceptant que les valeurs "always", "unless-stopped" et "on-failure". Utilisez-la dans tous les conteneurs de services.tf.
  3. Dans locals.tf, la variable db_url est construite dynamiquement. Expliquez pourquoi une référence à docker_container.postgres.name crée une dépendance implicite. Vérifiez-le dans le graphe de dépendances (terraform graph).
  4. Ajoutez un local all_service_ports qui construit une map associant le nom de chaque service à son port hôte. Utilisez-le dans un output services_ports.

1. sensitive = true : Dans le plan, la valeur est masquée avec (sensitive value) même quand elle change. Terraform cache la valeur dans tous les logs, mais elle reste présente en clair dans le state.

2. Variable restart_policy :

variable "restart_policy" {
  type    = string
  default = "unless-stopped"
  validation {
    condition     = contains(["always", "unless-stopped", "on-failure"], var.restart_policy)
    error_message = "Valeurs acceptées : always, unless-stopped, on-failure."
  }
}

Utilisée dans tous les conteneurs avec restart = var.restart_policy.

3. Dépendance implicite via locals : local.db_url référence docker_container.postgres.name → Terraform sait qu'il doit créer postgres avant d'évaluer db_url. Tout conteneur utilisant local.db_url dépend implicitement de postgres. Visible dans terraform graph.

4. all_service_ports : {frontend=8080, user_service=3001, product_service=3002, order_service=3003}. Output services_ports = local.all_service_ports exporté après apply.

08Dépendances implicites et explicites

  2. Supprimez le bloc depends_on de order-service. Lancez terraform plan puis terraform graph. Le comportement change-t-il ?
  3. Créez un scénario où un depends_on explicite est indispensable — c'est-à-dire où aucune référence directe ne crée la dépendance nécessaire. Décrivez ce scénario.

1. depends_on redondant : Non, pas nécessaire. La dépendance implicite existe via local.db_url → docker_container.postgres.name ET via les env vars USER_SERVICE_URL/PRODUCT_SERVICE_URL qui référencent docker_container.user_service.name et product_service.name.

2. Sans depends_on : terraform graph produit le même résultat, les arcs implicites couvrent déjà toutes les dépendances. Le comportement est identique.

3. Scénario indispensable : Un provisioner "local-exec" sur un conteneur qui exécute des migrations SQL en se connectant à postgres, mais sans référencer aucun attribut de postgres dans sa configuration HCL. Aucune dépendance implicite n'existe → depends_on = [docker_container.postgres] est indispensable.

11Cycle de vie et terraform taint
  1. Dans le module microservice, un bloc lifecycle { ignore_changes = [image] } est présent. Simulez le cas où l'image du user-service est mise à jour (modifiez le code, rebuildez). Que se passe-t-il lors du terraform plan avec ce bloc ? Sans ce bloc ?
  2. Utilisez terraform taint docker_container.user_service (ou son équivalent moderne). Que fait cette commande ? Vérifiez avec terraform plan.
  3. Ajoutez un bloc lifecycle { create_before_destroy = true } sur le conteneur postgres. Quel impact cela a-t-il sur la procédure de remplacement du conteneur ? Pourquoi est-ce utile pour un service avec état (stateful) ?

1. lifecycle ignore_changes = [image] : Avec le bloc → même si l'image change (nouveau sha256), Terraform ignore cet attribut → le conteneur n'est PAS recréé. Sans → Terraform propose de détruire et recréer le conteneur (remplacement forcé).

2. terraform taint / -replace : terraform apply -replace=docker_container.user_service marque la ressource pour recréation inconditionnelle. terraform plan affiche « must be replaced » même sans changement de configuration.

3. create_before_destroy sur postgres : Terraform crée le nouveau conteneur AVANT de détruire l'ancien → le volume reste monté, le service reste disponible pendant la transition. Sans ce bloc : postgres est détruit puis recréé → downtime. Indispensable pour les services stateful.

Partie 4 — count, for_each et expressions avancées

12count vs for_each
  1. Actuellement, chaque conteneur de service est une ressource distincte dans services.tf. Réécrivez les trois conteneurs de microservices (user, product, order) en utilisant une seule ressource docker_container avec for_each itérant sur le local services défini dans locals.tf.
  2. Quelle est la différence dans le state entre une ressource créée avec count = 3 et une ressource créée avec for_each sur un map de 3 éléments ? Illustrez avec les adresses de ressources Terraform.
  3. Vous retirez un service du map for_each. Que se passe-t-il ? Même question avec count si vous retirez un élément au milieu de la liste.
  4. La variable d'environnement DATABASE_URL est commune à tous les services, mais USER_SERVICE_URL et PRODUCT_SERVICE_URL sont spécifiques à l'order-service. Comment gérez-vous cette différence dans votre ressource for_each ?

1. Réécriture avec for_each :

resource "docker_container" "services" {
  for_each = local.services
  name     = "${var.project_name}_${each.key}_service"
  image    = docker_image.services[each.key].image_id
  restart  = var.restart_policy
  env = concat(
    ["PORT=${each.value.port}", "DATABASE_URL=${local.db_url}"],
    each.key == "order" ? [
      "USER_SERVICE_URL=http://...:3001",
      "PRODUCT_SERVICE_URL=http://...:3002"
    ] : []
  )
}

2. Différence count vs for_each dans le state :
count=3 → adresses docker_container.services[0], [1], [2] (index numérique).
for_each → adresses docker_container.services["user"], ["product"], ["order"] (clé de map).

3. Suppression : Avec for_each → seul l'élément retiré est détruit, les autres non affectés. Avec count → retirer un élément au milieu réindexe les suivants → Terraform détruit et recrée les éléments réindexés (destructif).

4. Env spécifiques à order : Utilisation de concat() + condition each.key == "order" pour ajouter USER_SERVICE_URL et PRODUCT_SERVICE_URL uniquement pour l'order-service.

13Expressions : fonctions et conditions
  1. Ajoutez une variable booléenne expose_postgres (défaut : true). Utilisez une expression conditionnelle dans le bloc ports du conteneur postgres pour n'exposer le port 5432 sur l'hôte que si cette variable est true. Vérifiez les deux cas avec terraform plan.
  2. Dans images.tf, le hash de déclenchement de rebuild est construit avec sha256(join(...)). Expliquez le rôle de ce mécanisme. Que se passe-t-il si vous modifiez src/index.js du user-service et relancez terraform apply ?
  3. Utilisez la fonction toset() pour créer un volume Docker par service (user, product, order) en une seule ressource docker_volume avec for_each. Ces volumes resteront vides pour l'instant.

1. expose_postgres conditionnel avec dynamic :

dynamic "ports" {
  for_each = var.expose_postgres ? [1] : []
  content {
    internal = 5432
    external = var.postgres_port
  }
}

Avec expose_postgres=false : terraform plan montre la suppression du bloc ports → port 5432 inaccessible depuis l'hôte.

2. sha256(join(...)) : Hash des contenus sources. Si un fichier change, le hash change → le champ triggers de docker_image change → Terraform rebuild l'image. En modifiant src/index.js : trigger mis à jour → image reconstruit → conteneur recréé.

3. toset() pour volumes :

resource "docker_volume" "service_volumes" {
  for_each = toset(["user", "product", "order"])
  name     = "${var.project_name}_${each.key}_data"
  labels { label = "service"; value = each.key }
}

Adresses Terraform : docker_volume.service_volumes["user"], ["product"], ["order"].

Module service-stack créé dans modules/service-stack/ :

main.tf : docker_image avec triggers filesha256, docker_container avec lifecycle { ignore_changes = [image] }, docker_volume avec count = var.enable_volume ? 1 : 0.

module "user_service" {
  source        = "./modules/service-stack"
  name          = "user-service"
  build_context = "${path.root}/../services/user-service"
  internal_port = 3001
  external_port = var.user_service_port
  network_name  = docker_network.app.name
  env_vars      = ["PORT=3001", "DATABASE_URL=${local.db_url}"]
  project_name  = var.project_name
  enable_volume = false
}

Partie 6 — Mise à jour, destroy et cas avancés

16Mise à jour d'un service sans interruption
  1. Modifiez le user-service : ajoutez un champ role (défaut : "user") dans la route POST /users et renvoyez-le dans les réponses GET. Vérifiez localement que la modification fonctionne.
  2. Mettez à jour le tag de l'image en changeant services_image_tag à "v2" dans terraform.tfvars. Lancez terraform plan : quelles ressources sont affectées ? Pourquoi plus que juste le conteneur ?
  3. Avec le lifecycle { ignore_changes = [image] } actif dans le module, expliquez pourquoi le conteneur ne sera pas recréé même si l'image change. Comment forcer la recréation ?
  4. Après l'apply, vérifiez que la nouvelle version du service répond correctement. Utilisez docker inspect pour confirmer quelle image est utilisée par le conteneur.

1. Ajout champ role : Schéma SQL modifié (colonne role VARCHAR(50) DEFAULT 'user'), POST /users accepte role dans le body, routes GET retournent le champ role.

2. services_image_tag = « v2 » : Le nom de l'image change → le trigger sha256 change → docker_image est recréé → image_id change → le conteneur est marqué pour remplacement. Plus que juste le conteneur : l'image est aussi reconstruite.

3. lifecycle ignore_changes = [image] : Terraform ignore les modifications de l'attribut image → le conteneur tourne encore avec l'ancienne image malgré le rebuild. Pour forcer : terraform apply -replace=docker_container.user_service.

4. Vérification : curl http://localhost:3001/users → champ role présent. docker inspect shopmicrfo_user_service | grep Image → confirme le sha256 de l'image active.

17Gestion de la destruction et des données persistantes
  1. Insérez quelques utilisateurs et produits dans l'application. Lancez terraform destroy. Que devient le volume postgres_data ? Les données sont-elles perdues ?
  2. Relancez terraform apply. Les données sont-elles restaurées ? Expliquez en fonction du comportement du provider Docker sur les volumes.
  3. Ajoutez un bloc lifecycle { prevent_destroy = true } sur la ressource docker_volume.postgres_data. Testez terraform destroy. Que se passe-t-il ? Comment contourner cette protection quand c'est vraiment nécessaire ?
  4. Supprimez uniquement les conteneurs sans toucher au volume ni aux images. Écrivez la ou les commandes terraform permettant de cibler des ressources spécifiques.

1. terraform destroy + prevent_destroy : Sans le bloc lifecycle, le volume postgres_data est supprimé par le provider Docker et les données sont perdues. Avec lifecycle { prevent_destroy = true } : Terraform refuse et affiche une erreur.

2. terraform apply après destroy : Si le volume existe encore (prevent_destroy ou conservé manuellement), PostgreSQL retrouve ses fichiers dans /var/lib/postgresql/data → données restaurées. Le provider ne supprime pas les volumes nommés sauf si force_destroy = true.

3. Contournement : Commenter temporairement le bloc lifecycle, appliquer les changements, puis relancer destroy. Ou modifier force_destroy = true.

4. Cibler uniquement les conteneurs :

  -target=docker_container.user_service \

18Provisioners et scripts d'initialisation
  1. Consultez la documentation Terraform sur les provisioners. Quelle est la philosophie officielle de Terraform à leur égard ?
  2. Ajoutez un provisioner local-exec sur le conteneur postgres qui attend que la base soit prête avant de continuer (utilisez docker exec ... pg_isready). Ce provisioner doit s'exécuter à la création uniquement.
  3. Testez le comportement : détruisez le conteneur postgres et relancez terraform apply. Le provisioner est-il exécuté lors de la recréation ?
  4. Pourquoi les provisioners sont-ils considérés comme un "last resort" en Terraform ? Proposez une alternative pour l'initialisation de schéma de base de données.

1. Philosophie Terraform : Les provisioners sont un « last resort » : Terraform ne peut pas tracker leur exécution dans le state → non idempotent. En cas d'échec, la ressource est marquée « tainted ». Préférer les ressources natives du provider.

2. local-exec sur postgres :

provisioner "local-exec" {
  command = "until docker exec ${self.name} pg_isready -U ${var.db_user} -d ${var.db_name}; do sleep 2; done"
}

S'exécute uniquement à la création (comportement par défaut, pas sur destroy).

3. Lors de la recréation : Oui, le provisioner est réexécuté car c'est une nouvelle création de ressource (destroy + create déclenche le provisioner).

4. Alternatives : Utiliser un outil de migration dédié (Flyway, Liquibase) déclenché par le pipeline CI/CD, ou un init container exécutant les migrations avant les services. Avantages : idempotence, versionning du schéma, rollback possible.

Concepts clés

State Le fichier terraform.tfstate

Terraform stocke l'état réel de l'infrastructure dans un fichier JSON appelé state. Il contient les IDs, IPs et attributs de chaque ressource après création. C'est la source de vérité de Terraform : lors d'un plan, Terraform compare le code avec le state pour savoir ce qui a changé.

Ne jamais modifier le state à la main. Utiliser terraform state mv, terraform state rm ou terraform import selon le besoin.

En équipe, le state doit être partagé via un backend distant (HTTP, S3, Terraform Cloud…) pour que tous les membres voient le même état. Un mécanisme de verrou empêche deux personnes d'appliquer simultanément.

Dérive Drift — quand la réalité s'écarte du code

Une dérive se produit quand une ressource est modifiée ou supprimée manuellement hors de Terraform (via l'interface Docker, la console cloud…). Au prochain terraform plan, Terraform détecte la différence entre le state et la réalité et propose de la corriger.

Exemple : si on supprime un conteneur avec docker rm, Terraform le recréera au prochain apply pour revenir à l'état désiré.

Dépendances Implicites vs explicites

Implicite : Quand une ressource A référence un attribut d'une ressource B (docker_network.app.name), Terraform infère automatiquement que B doit être créée avant A. C'est la forme la plus courante et recommandée.

Explicite : Le bloc depends_on = [resource.B] force un ordre quand il n'y a aucune référence directe. À utiliser seulement quand nécessaire, car il rend le graphe plus difficile à lire.

count / for_each Créer plusieurs ressources similaires

count = N — crée N instances identiques d'une ressource, accessibles via resource.name[0], [1]… Fragile : si on retire un élément au milieu de la liste, Terraform recréera toutes les ressources suivantes.

for_each = map — crée une instance par entrée d'un map ou d'un set. Chaque instance est adressée par sa clé (resource.name["clé"]). Stable : retirer une entrée ne touche que la ressource correspondante.

Règle : Toujours préférer for_each sur des maps nommés dès que possible.

lifecycle Contrôler le comportement lors des mises à jour
  • ignore_changes = [attr] — Terraform ignore les différences sur cet attribut. Utile quand Docker ou un autre outil modifie un attribut hors du contrôle de Terraform (ex : gateway assignée automatiquement).
  • prevent_destroy = true — Terraform refuse de détruire cette ressource, même lors d'un terraform destroy. Protection contre les suppressions accidentelles (volumes de données, bases de production).
  • create_before_destroy = true — Terraform crée le remplaçant avant de détruire l'ancien. Utile pour les services avec état où une interruption serait problématique.
Modules Encapsuler et réutiliser

Un module est un dossier de fichiers .tf exposant des variables (inputs) et des outputs. Il s'appelle depuis un autre fichier avec module "nom" { source = "./chemin" }.

Un module bien conçu est autonome, documenté (descriptions des variables), et ne contient pas de valeurs en dur. Il doit pouvoir être instancié plusieurs fois avec des paramètres différents.

Le Terraform Registry (registry.terraform.io) propose des modules communautaires maintenus pour les fournisseurs courants (AWS, Docker, Kubernetes…).

Variables Types, valeurs par défaut, validation

Les variables Terraform sont déclarées dans variables.tf. Leur valeur peut être passée par :

  • Fichier terraform.tfvars (chargé automatiquement)
  • Flag -var="nom=valeur" sur la ligne de commande
  • Variable d'environnement TF_VAR_nom
  • Valeur par défaut dans la déclaration (default = ...)

Le bloc validation permet de rejeter des valeurs invalides avec un message d'erreur explicite, avant même que Terraform contacte l'infrastructure.

Les variables sensitive = true ne sont jamais affichées dans les logs de plan/apply.

Locals Valeurs calculées réutilisables

Les locals (déclarés dans un bloc locals { }) permettent de nommer des expressions complexes pour les réutiliser sans répétition. Contrairement aux variables, ils ne sont pas exposés à l'extérieur du module et ne peuvent pas être surchargés depuis la ligne de commande.

Exemple typique : construire une URL de connexion à partir de plusieurs variables (host, port, user), puis référencer ce local dans plusieurs ressources.

20Analyse de sécurité du code Terraform

Relisez l'ensemble du code Terraform produit pendant le TP et répondez aux questions suivantes.

  1. Le mot de passe PostgreSQL est déclaré comme variable sensitive = true et stocké dans terraform.tfvars. Ouvrez le fichier terraform.tfstate après un apply. Le mot de passe y est-il présent ? En clair ou chiffré ? Quelle conclusion en tirez-vous sur la protection du state ?

  3. Dans images.tf, le trigger de rebuild est calculé avec sha256(join(...)) sur trois fichiers. Quel fichier important lié aux dépendances Node.js n'est pas inclus dans ce calcul ? Quelles conséquences cela peut-il avoir ? Corrigez le trigger.
  4. La variable db_password a la valeur par défaut "shoppassword" dans terraform.tfvars. Proposez un mécanisme Terraform permettant d'obliger l'opérateur à fournir explicitement un mot de passe en production, sans valeur par défaut possible.

1. terraform.tfstate et mot de passe : Oui, le mot de passe est présent en clair dans le state, dans les attributs env des conteneurs utilisant DATABASE_URL. sensitive = true masque la valeur dans les logs/plan mais PAS dans le state. Conclusion : le state doit être chiffré (backend S3 avec SSE, Terraform Cloud) et ne jamais être commité en git.

2. Port 5432 exposé : Oui, depuis la machine hôte : psql -h localhost -p 5432 -U shopuser -d shopdb fonctionne.
