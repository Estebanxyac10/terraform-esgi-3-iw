Contexte & Architecture cible

Le projet ShopMicro est une application e-commerce décomposée en quatre services indépendants. Votre objectif est de déployer et gérer cette architecture localement en utilisant Terraform avec le provider Docker, sans toucher à Docker Compose une fois la Partie 1 terminée.

Concepts clés

State Le fichier terraform.tfstate

Terraform stocke l'état réel de l'infrastructure dans un fichier JSON appelé state. Il contient les IDs, IPs et attributs de chaque ressource après création. C'est la source de vérité de Terraform : lors d'un plan, Terraform compare le code avec le state pour savoir ce qui a changé.

Ne jamais modifier le state à la main. Utiliser terraform state mv, terraform state rm ou terraform import selon le besoin.

En équipe, le state doit être partagé via un backend distant (HTTP, S3, Terraform Cloud…) pour que tous les membres voient le même état. Un mécanisme de verrou empêche deux personnes d'appliquer simultanément.

Dérive Drift — quand la réalité s'écarte du code

Une dérive se produit quand une ressource est modifiée ou supprimée manuellement hors de Terraform (via l'interface Docker, la console cloud…). Au prochain terraform plan, Terraform détecte la différence entre le state et la réalité et propose de la corriger.

Exemple : si on supprime un conteneur avec docker rm, Terraform le recréera au prochain apply pour revenir à l'état désiré.

Dépendances Implicites vs explicites

Implicite : Quand une ressource A référence un attribut d'une ressource B (docker_network.app.name), Terraform infère automatiquement que B doit être créée avant A. C'est la forme la plus courante et recommandée.

Explicite : Le bloc depends_on = [resource.B] force un ordre quand il n'y a aucune référence directe. À utiliser seulement quand nécessaire, car il rend le graphe plus difficile à lire.

count / for_each Créer plusieurs ressources similaires

count = N — crée N instances identiques d'une ressource, accessibles via resource.name[0], [1]… Fragile : si on retire un élément au milieu de la liste, Terraform recréera toutes les ressources suivantes.

for_each = map — crée une instance par entrée d'un map ou d'un set. Chaque instance est adressée par sa clé (resource.name["clé"]). Stable : retirer une entrée ne touche que la ressource correspondante.

Règle : Toujours préférer for_each sur des maps nommés dès que possible.

lifecycle Contrôler le comportement lors des mises à jour
  • ignore_changes = [attr] — Terraform ignore les différences sur cet attribut. Utile quand Docker ou un autre outil modifie un attribut hors du contrôle de Terraform (ex : gateway assignée automatiquement).
  • prevent_destroy = true — Terraform refuse de détruire cette ressource, même lors d'un terraform destroy. Protection contre les suppressions accidentelles (volumes de données, bases de production).
  • create_before_destroy = true — Terraform crée le remplaçant avant de détruire l'ancien. Utile pour les services avec état où une interruption serait problématique.
Modules Encapsuler et réutiliser

Un module est un dossier de fichiers .tf exposant des variables (inputs) et des outputs. Il s'appelle depuis un autre fichier avec module "nom" { source = "./chemin" }.

Un module bien conçu est autonome, documenté (descriptions des variables), et ne contient pas de valeurs en dur. Il doit pouvoir être instancié plusieurs fois avec des paramètres différents.

Le Terraform Registry (registry.terraform.io) propose des modules communautaires maintenus pour les fournisseurs courants (AWS, Docker, Kubernetes…).

Variables Types, valeurs par défaut, validation

Les variables Terraform sont déclarées dans variables.tf. Leur valeur peut être passée par :

  • Fichier terraform.tfvars (chargé automatiquement)
  • Flag -var="nom=valeur" sur la ligne de commande
  • Variable d'environnement TF_VAR_nom
  • Valeur par défaut dans la déclaration (default = ...)

Le bloc validation permet de rejeter des valeurs invalides avec un message d'erreur explicite, avant même que Terraform contacte l'infrastructure.

Les variables sensitive = true ne sont jamais affichées dans les logs de plan/apply.

Locals Valeurs calculées réutilisables

Les locals (déclarés dans un bloc locals { }) permettent de nommer des expressions complexes pour les réutiliser sans répétition. Contrairement aux variables, ils ne sont pas exposés à l'extérieur du module et ne peuvent pas être surchargés depuis la ligne de commande.

Exemple typique : construire une URL de connexion à partir de plusieurs variables (host, port, user), puis référencer ce local dans plusieurs ressources.

19Nouveau microservice : notification-service

  • GET /notifications/:user_id — retourne les notifications d'un utilisateur donné.
  1. Application : Créez le dossier services/notification-service/ avec son src/index.js, son package.json, son Dockerfile et son package-lock.json. Le service doit fonctionner en mode mémoire (sans DATABASE_URL) et en mode PostgreSQL.

Pensez à : l'image Docker, le conteneur, la variable de port, l'output.   4. Test de bout en bout : Via curl, créez un utilisateur, envoyez deux notifications à cet utilisateur, puis récupérez uniquement ses notifications. Montrez les commandes et réponses.
  5. Dérive volontaire : Après l'apply, renommez manuellement le conteneur notification-service en modifiant son label avec docker rename. Lancez terraform plan. Que se passe-t-il ? Pourquoi Terraform ne détecte-t-il pas le changement de nom ? Quelle commande permet de forcer la resynchronisation ?

Proposez une architecture alternative pour centraliser la gestion du schéma de base de données. Vous n'avez pas à l'implémenter, mais décrivez la solution et ses avantages.
